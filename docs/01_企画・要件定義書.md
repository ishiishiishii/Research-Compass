## 1. プロジェクト概要

### 1.1 一言説明

**Research Compass** は、論文・研究手法を「地図（グラフ）」として可視化し、個人の学習メモ・理解度を管理し、ゼミメンバー同士で各自の論文図を閲覧できる Web アプリ。

### 1.2 コンセプト

> **「論文を探すな、地図を作れ」**

論文検索ツールではなく、**自分（またはゼミ）の研究学習を可視化・共有する Research Navigation System** を目指す。

### 1.3 今回のスコープ変更（重要）

| 項目 | 当初案 | **MVP（今回）** |
|------|--------|-----------------|
| 分野入力 → 自動ロードマップ生成 | メイン機能 | **却下**（将来機能） |
| 論文・手法のグラフ管理 | サブ | **メイン機能** |
| 理解メモ・理解度管理 | メイン | **メイン機能** |
| グループ内メンバーの論文図閲覧 | なし | **メイン機能** |
| グループ統合グラフの自動生成 | 検討中 | **却下** |

## 2. 背景・課題（Why）

### 2.1 きっかけ（実体験）

ゼミの論文発表ゼミでは、Google ドライブ上で分野別フォルダに PDF や資料を保管していた。  
しかし以下の問題があった。

- フォルダ分けだけでは**論文間の関連性**が見えない
- 誰が何を読んだか把握しづらい
- メモがファイルごとに散在し、**後から振り返りにくい**
- 同じ論文を別メンバーが重複して読むことがある

### 2.2 解決したい課題

| # | 課題 | Research Compass での解決 |
|---|------|---------------------------|
| 1 | 論文・手法の関係が把握しづらい | グラフ（ノード＋矢印）で可視化 |
| 2 | 学習メモが散在する | 各ノードにメモ・理解度を紐づけ |
| 3 | 読んだ論文の管理が難しい | 関連/非関連トグルで重要度を調整しつつ履歴を保持 |
| 4 | ゼミ内の読了状況が見えない | グループ機能でメンバー各自の論文図を閲覧 |

### 2.3 対象ユーザー

| ユーザー | ニーズ |
|----------|--------|
| 研究室配属直後の学生 | 分野の論文を体系的に整理したい |
| 論文発表ゼミの学生 | ゼミメンバーの読了状況・関連図を参照したい |
| 新分野を学び始める研究者 | 自分だけの学習地図を作りたい |

## 3. プロダクトゴール

### 3.1 MVP の成功基準

- [ ] ユーザーが論文・手法ノードを追加し、グラフ上で関係を結べる
- [ ] ノードをクリックして詳細・メモ・理解度を保存・閲覧できる
- [ ] 関連/非関連を切り替え、非関連ノードは薄く表示できる
- [ ] グループを作成し、メンバー各自の論文図を**いつでも閲覧**できる
- [ ] デモシナリオ（§13）を 5 分以内に完走できる

### 3.2 非ゴール（MVP ではやらない）

- 分野名入力による自動ロードマップ生成
- 論文 PDF の自動解析・要約
- グループ全体の統合グラフ自動生成
- AI 質問応答
- スマホ最適化（PC 優先）

### 4.2 F-02：個人論文グラフ管理

#### 概要
ユーザーが自分の研究地図（有向グラフ）を作成・編集する。

#### ノード（論文/手法）に含める情報

| 項目 | 必須 | 例 |
|------|------|-----|
| タイトル / 手法名 | ○ | DQN |
| 発表年 | △ | 2015 |
| 概要・要約 | △ | Deep Q-Network の概要 |
| 解決した課題 | △ | 高次元状態での Q 学習 |
| 貢献 | △ | CNN による Q 関数近似 |
| 論文 URL | △ | https://... |
| 前提知識（タグ） | △ | Bellman Equation, Replay Buffer |
| 関連フラグ | ○ | 関連 / 非関連 |
| 理解度 | △ | ◎ / ○ / △ / × / 未設定 |

#### エッジ（関係）

| 項目 | 説明 | 例 |
|------|------|-----|
| from → to | 依存・発展・参考関係 | Q-learning → DQN |
| ラベル（任意） | 関係の種類 | improves, prerequisite |

#### 操作

- ノード追加（手動入力）
- ノード編集・削除
- エッジ追加（ドラッグ or 選択式）
- エッジ削除
- グラフのパン・ズーム

### 4.3 F-03：ノード詳細・メモ

#### 概要
グラフ上のノードをクリックするとサイドパネルまたはモーダルで詳細を表示。

#### 理解メモ

| 項目 | 説明 |
|------|------|
| 理解できなかった点 | テキスト（例：Target Network の役割） |
| 自由メモ | 任意のテキスト |
| 更新日時 | 自動記録 |

---

### 4.4 F-04：関連/非関連トグル

#### 概要
ノードごとに「自分の分野と関連するか」を切り替える。

#### 挙動

| 状態 | 表示 | 目的 |
|------|------|------|
| 関連 | 通常表示（opacity: 1.0） | 今の学習の中心 |
| 非関連 | 薄く表示（opacity: 0.3〜0.4） | 読んだ履歴は残し、重要度を下げる |

- 非関連ノードも**削除せず保持** → 重複読み防止の目印になる
- フィルタ：「関連のみ表示」トグル（任意）

---

### 4.5 F-05 / F-06：グループ機能

#### 概要
ゼミ単位などのグループを作成し、メンバー各自の論文図を閲覧できる。

#### グループ機能詳細

| 操作 | 説明 |
|------|------|
| グループ作成 | 名前・説明を入力して作成 |
| メンバー招待 | 招待コード or リンクで参加 |
| メンバー一覧 | グループ内メンバーを表示 |
| 論文図閲覧 | メンバーを選択 → **その人のグラフを閲覧（読み取り専用）** |

#### 意図（ゼミ向け）

- ドライブのフォルダ分けの代わりに、**各自の論文地図**を一覧から選んで見られる
- 「A さんは DQN 周辺を読んでいる」「B さんは Policy Gradient 系」が一目でわかる
- 統合グラフは作らない → 実装コスト削減、レイアウト崩れ・マージ精度の問題を回避

#### 閲覧モードの制約

- 他メンバーのグラフ：**閲覧のみ**（編集不可）
- 自分のグラフ：編集可能

## 6. データモデル

### 6.1 ER 図（概念）

```
users ─────┬───── nodes ───── edges
           │         │
           │         ├── memos
           │         └── (understanding, is_relevant は nodes に含む)
           │
           ├──── group_members ───── groups
           │
           └── (各 user が独自の nodes / edges を持つ)
```

### 6.2 テーブル定義

#### users

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| email | string | ログイン用 |
| display_name | string | 表示名 |
| created_at | datetime | 作成日時 |

#### groups

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| name | string | グループ名（例：〇〇ゼミ 2026） |
| description | string | 説明 |
| invite_code | string | 招待コード |
| created_by | UUID | FK → users.id |
| created_at | datetime | |

#### group_members

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| group_id | UUID | FK → groups.id |
| user_id | UUID | FK → users.id |
| joined_at | datetime | |

#### nodes

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |
| user_id | UUID | FK → users.id（所有者） |

| カラム | 型 | 説明 |

## 7. API 設計（概要）

REST API（FastAPI）または Supabase Client 直接呼び出し。

| Method | Endpoint | 説明 |
|--------|----------|------|
| POST | `/auth/register` | 新規登録 |
| POST | `/auth/login` | ログイン |
| GET | `/nodes` | 自分のノード一覧 |
| POST | `/nodes` | ノード作成 |
| PUT | `/nodes/:id` | ノード更新 |
| DELETE | `/nodes/:id` | ノード削除 |
| GET | `/edges` | 自分のエッジ一覧 |
| POST | `/edges` | エッジ作成 |
| DELETE | `/edges/:id` | エッジ削除 |
| GET | `/groups` | 所属グループ一覧 |
| POST | `/groups` | グループ作成 |
| POST | `/groups/:id/join` | 招待コードで参加 |
| GET | `/groups/:id/members` | メンバー一覧 |
| GET | `/users/:id/nodes` | 指定ユーザーのノード（グループ内閲覧用） |
| GET | `/users/:id/edges` | 指定ユーザーのエッジ（グループ内閲覧用） |

**アクセス制御（F-06）：**  
`/users/:id/nodes` は、リクエスト者と対象ユーザーが**同一グループに所属**している場合のみ許可。

## 9. 技術選定

### 9.1 技術スタック（確定案）

```
┌─────────────────────────────────────────────┐
│  Frontend                                   │
│  React 18 + TypeScript + Vite               │
│  Tailwind CSS                               │
│  React Flow（グラフ可視化）                  │
│  React Router                               │
├─────────────────────────────────────────────┤
│  Backend                                    │
│  Supabase（Auth + PostgreSQL + REST API）   │
│  ※ 必要に応じて Edge Functions              │
├─────────────────────────────────────────────┤
│  Hosting                                    │
│  Vercel（Frontend）+ Supabase Cloud（BaaS） │
└─────────────────────────────────────────────┘
```

### 9.2 各技術の選定理由

#### Frontend：React + TypeScript + Vite

| 観点 | 理由 |
|------|------|
| 選定理由 | グラフ UI ライブラリ（React Flow）が React 前提。TypeScript で型安全に開発。Vite は起動・HMR が速く 1 週間開発向き |
| React Flow | 本アプリの核心が「ノード＋エッジのグラフ」。D3 や Cytoscape より React との統合が容易 |

**他案との比較**

| 代替 | 不採用理由 |
|------|------------|
| Vue / Svelte | React Flow が React 専用。代替の vue-flow 等は情報量・実績で React Flow に劣る |
| Next.js | SSR は本アプリに不要。Vite の方がセットアップが軽く、初参加ハッカソン向き |
| Create React App | 非推奨化。Vite の方が高速 |

#### UI：Tailwind CSS

| 観点 | 理由 |
|------|------|
| 選定理由 | ユーティリティクラスで UI を素早く組める。デザインシステム構築時間を削減 |

**他案との比較**

| 代替 | 不採用理由 |
|------|------------|
| Material UI / Chakra UI | 導入は楽だが、グラフ画面との一体感調整に時間がかかる |
| 素の CSS | 1 週間では UI 作成に時間を取られすぎる |

#### Backend / DB：Supabase

| 観点 | 理由 |
|------|------|
| 選定理由 | **認証・PostgreSQL・REST API が一体化**。グループ機能（マルチユーザー）に必須の Auth を数時間で導入可能。無料枠あり |
| Row Level Security | 「同一グループメンバーのみ閲覧可」を DB ポリシーで enforce 可能 |

**他案との比較**

| 代替 | 不採用理由 |
|------|------------|
| FastAPI + SQLite | 認証・グループ ACL を自前実装する時間が 1 週間では足りない |
| Firebase | NoSQL は nodes/edges の relational クエリが冗長。Supabase（PostgreSQL）の方が ER モデルに合う |
| FastAPI + PostgreSQL（自前） | DB・Auth・デプロイのセットアップコストが Supabase より大 |
| MongoDB | グラフ構造は RDB + JSON カラムで十分。学習コスト不要 |

#### Hosting：Vercel + Supabase Cloud

| 観点 | 理由 |
|------|------|
| 選定理由 | 無料枠、GitHub 連携デプロイ、HTTPS 自動。ハッカソン発表 URL を即日発行可能 |

**他案との比較**

| 代替 | 不採用理由 |
|------|------------|
| ローカルのみ | 発表・審査で URL デモができない |
| AWS EC2 | セットアップ・コスト管理が重い |

#### 却下した技術（MVP 範囲）

| 技術 | 当初案での用途 | 却下理由 |
|------|----------------|----------|
| sentence-transformers | 論文類似検索 | 自動サジェスト機能却下 |
| PyMuPDF | PDF 解析 | MVP 外 |
| NetworkX（サーバー側） | グラフ生成 | フロントの React Flow で十分。自動生成しない |
| OpenAI / LLM API | ロードマップ生成 | コスト・精度・時間の問題 |

## 10. システムアーキテクチャ

```
[ Browser ]
    │
    │  HTTPS
    ▼
[ Vercel ] ─── React App（React Flow UI）
    │
    │  Supabase Client (JS)
    ▼
[ Supabase ]
    ├── Auth（JWT）
    ├── PostgreSQL（nodes, edges, groups...）
    └── Row Level Security（グループ閲覧制御）
```

### 10.1 グループ閲覧の権限制御（要点）

```
リクエスト: user A が user B の nodes を取得

1. A と B が同一 group_members に存在するか確認
2. YES → 取得許可（READ ONLY）
3. NO  → 403 Forbidden
```

## 13. ハッカソン発表デモシナリオ（5 分）

### ストーリー

> 「ゼミでは Google ドライブで分野別に論文を管理していましたが、関連性が見えませんでした。Research Compass なら、各自が論文地図を作り、ゼミメンバー同士で共有・閲覧できます。」

### デモ手順

1. **ログイン**（事前作成アカウント）
2. **マイ論文グラフ**を表示（強化学習シードデータ：Q-learning → DQN → PPO）
3. **DQN ノードをクリック** → 詳細・メモ・理解度「○」を表示
4. **非関連ノード**（DDPG）を薄く表示 → トグル操作
5. **グループ画面** → 「〇〇ゼミ 2026」を開く
6. **メンバー「田中」を選択** → 田中さんの論文図（Policy Gradient 系）を閲覧
7. **クロージング**：「ドライブのフォルダの代わりに、生きた研究地図を」
