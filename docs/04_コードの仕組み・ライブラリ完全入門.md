### 0.1 ファイルの拡張子 `.js` / `.ts` / `.jsx` / `.tsx` とは何か

パソコン上のファイル名の **末尾の `.〇〇`** は「このファイルは何の形式か」を表します。

| 拡張子 | 正式名称 | 中身は何か | このプロジェクトでの例 |
|--------|----------|------------|------------------------|
| `.js` | JavaScript | ブラウザや Node が実行できるプログラム | ビルド後の `dist/assets/index-xxxxx.js` |
| `.ts` | TypeScript | JS + 型（データの形の約束） | `lib/api/groups.ts`, `types/index.ts` |
| `.jsx` | JavaScript + JSX | HTML 風の記法を含む JS | （ほぼ使わない。tsx に統一） |
| `.tsx` | TypeScript + JSX | 型付き + HTML 風の記法 | `GroupsPage.tsx`, `App.tsx` |

#### ブラウザは `.tsx` をそのまま実行できない

これが非常に重要です。

```
あなたが書くコード          ブラウザが実際に実行するもの
─────────────────          ───────────────────────────
main.tsx      ──変換──→    main.js（1 つにまとめられた JS）
GroupsPage.tsx ──変換──→   （上に含まれる）
graph-utils.ts ──変換──→   （上に含まれる）
```

- **ブラウザが理解できるのは基本的に JavaScript だけ**
- TypeScript の型（`: string` や `type Group`）は **変換のときに消える**（実行時には存在しない）
- JSX（`<h1>...</h1>`）も **普通の JS 関数呼び出しに変換される**

変換を担当するのが **Vite**（開発時はその場で、本番用は `npm run build` で）です。

#### `.tsx` ファイルの中身は 2 種類が混ざっている

```tsx
// ① 普通の TypeScript（ロジック）
const [name, setName] = useState('')
async function load() { ... }

// ② JSX（見た目の declarative な記述）
return (
  <section>
    <h1>グループ</h1>
  </section>
)
```

| 部分 | 役割 |
|------|------|
| 関数の上側 | データの準備、イベント処理、計算 |
| `return (...)` の中 | 画面に何を表示するかの設計図 |

**なぜ `.tsx` と `.ts` を分けるのか**

| 種類 | 使う場面 |
|------|----------|
| `.tsx` | JSX（`<div>` など）を書くファイル。画面コンポーネント |
| `.ts` | JSX を書かないファイル。API 関数、型定義、ユーティリティ |

例: `lib/api/groups.ts` は HTML を書かないので `.ts` だけで十分。  
`GroupsPage.tsx` は画面を描くので `.tsx` が必要です。

#### JSX が JS に変換されると（イメージ）

開発者が書く:

```tsx
<h1 className="text-2xl">グループ</h1>
```

変換後はだいたいこういう JS になります（概念図）:

```javascript
// React が「h1 要素を作れ」と命令する関数呼び出しに変わる
React.createElement('h1', { className: 'text-2xl' }, 'グループ')
```

だから **JSX は「ただの HTML ファイル」ではなく、JavaScript の一部** です。

### 0.2 「開発サーバー」と「ビルド」とは何か

この 2 つは混同されやすいので、役割を分けて理解してください。

#### たとえ話

| 概念 | たとえ |
|------|--------|
| **開発サーバー** | 工房の作業台。作りながらすぐ試せる。本人用 |
| **ビルド** | 完成品を工場でパッケージ化。お客さん（本番環境）に届ける用 |

#### 開発サーバー（`npm run dev`）

```powershell
cd frontend
npm run dev
```

実行すると Vite が **あなたの PC 上で小さな Web サーバー** を起動します。

```
ブラウザ http://localhost:5173
        ↓ ファイルを要求
Vite 開発サーバー（あなたの PC 内）
        ↓ その場で .tsx → .js に変換して返す
ブラウザが JS を実行して画面表示
```

| 特徴 | 説明 |
|------|------|
| 起動場所 | 開発者の PC（または Docker コンテナ） |
| URL | 通常 `http://localhost:5173` |
| ホットリロード | ファイルを保存すると数秒でブラウザが自動更新 |
| 変換タイミング | **アクセスされたとき** にその場で変換（速い） |
| 誰が使うか | 開発者だけ。一般ユーザーはアクセスしない |

**なぜ開発サーバーが要るのか**  
ブラウザは `.tsx` を読めないので、**誰かが変換して配る係** が必要だからです。  
本番前の開発中は、その係を自分の PC 上で動かしています。

#### ビルド（`npm run build`）

```powershell
cd frontend
npm run build
```

実行すると `frontend/dist/` フォルダに **本番配信用のファイル一式** が生成されます。

```
frontend/src/ のたくさんの .tsx / .ts
        ↓ 型チェック（tsc）
        ↓ 1 つ（または少数）の .js にまとめる
        ↓ CSS も最適化
frontend/dist/
  ├── index.html
  ├── assets/index-xxxxx.js   ← 全部入りの JS
  └── assets/index-xxxxx.css  ← スタイル
```

| 特徴 | 説明 |
|------|------|
| 出力先 | `frontend/dist/` |
| 変換タイミング | **ビルドコマンドを打ったときに一括** |
| ファイル数 | 元は数十〜数百ファイル → 数ファイルに圧縮 |
| 誰が使うか | Render などのホスティングが `dist/` を世界中に配信 |

**なぜビルドが要るのか**

1. ユーザーのブラウザに **小さく速いファイル** を渡したい（全部バラで渡すと遅い）
2. TypeScript や JSX を **ブラウザが実行できる形に確定** させたい
3. 本番サーバーに Vite を常時動かす必要がなくなる（静的ファイルを置くだけでよい）

#### 開発サーバー vs ビルド — 一覧表

| | 開発サーバー (`npm run dev`) | ビルド (`npm run build`) |
|--|------------------------------|---------------------------|
| 目的 | 開発中に試す | 本番に公開する |
| コマンド | `npm run dev` | `npm run build` |
| 結果 | サーバーが動き続ける | `dist/` フォルダができる |
| 更新 | 保存するたび即反映 | ビルドのたびに作り直し |
| 本番 URL | 使わない | Render が `dist/` を配信 |

#### `npm run` とは

`package.json` の `scripts` に書かれたコマンドのショートカットです。

```json
"scripts": {
  "dev": "vite",
  "build": "tsc -b && vite build"
}
```

| コマンド | 実際にやること |
|----------|----------------|
| `npm run dev` | `vite` を実行 → 開発サーバー起動 |
| `npm run build` | まず `tsc` で型チェック → 問題なければ `vite build` |

### 0.3 「Group（グループ）」とは何か — このアプリでの意味

**注意**: ここでいう Group は、React の機能やプログラミングの「グループ化」とは **別物** です。  
Research Compass では **ゼミ・研究室・勉強会などの「共有の輪」** を指します。

#### 日常のたとえ

```
個人の論文マップ  …  各自のノート（本人だけ編集できる）
グループ          …  ゼミの共有棚（メンバーは互いのノートを“読める”）
```

Google ドライブでいう **「共有フォルダに招待されたメンバー」** に近いイメージです。  
ただし Research Compass では **ファイルを上げるのではなく、各自の論文グラフを閲覧できる** 点が違います。

#### Group でできること / できないこと

| できること | できないこと（MVP） |
|------------|---------------------|
| グループを作成する | グループ全員のグラフを 1 つに自動マージ |
| 招待コードで参加する | グループ内で他人のグラフを編集する |
| メンバー一覧を見る | |
| メンバー各自の論文マップを閲覧する | |
| グループ向けにフィード投稿する | |

#### データベースでは 2 つのテーブルで表現される

**① `groups` テーブル — グループそのもの**

| カラム | 意味 | 例 |
|--------|------|-----|
| `id` | グループの一意な ID | `a1b2c3...`（UUID） |
| `name` | 表示名 | `〇〇ゼミ 2026` |
| `description` | 説明（任意） | `強化学習ゼミ` |
| `invite_code` | 参加用コード | `SEMI2026` |
| `created_by` | 作成者のユーザー ID | |

**② `group_members` テーブル — 誰がどのグループにいるか**

| カラム | 意味 |
|--------|------|
| `group_id` | どのグループか |
| `user_id` | どのユーザーか |
| `joined_at` | いつ参加したか |

```
groups（ゼミ A）────┬──── group_members（田中）
                   ├──── group_members（佐藤）
                   └──── group_members（あなた）
```

1 人が複数グループに入ることも、1 グループに複数人が入ることもあります（多対多の関係）。

#### コード上の `Group` 型

TypeScript の `Group` は、DB の `groups` テーブルの 1 行の形を表します。

```50:57:frontend/src/types/index.ts
export type Group = {
  id: string
  name: string
  description: string | null
  invite_code: string
  created_by: string
  created_at: string
}
```

画面で `group.name` と書くとき、この型に沿ったオブジェクトを扱っています。

#### 招待コードの流れ（Group の世界での 1 シーン）

```
① ゼミ代表がグループを作成
   → groups に 1 行追加
   → invite_code が自動生成（例: A3F9B2C1）
   → 代表は group_members にも追加される

② 新メンバーが GroupsPage でコードを入力
   → join_group_by_invite_code（DB の関数）が実行
   → group_members に「あなた」の行が追加

③ あなたと田中が同じ group_members にいる
   → RLS により、田中の nodes / edges を読める
   → MemberGraphPage で閲覧可能
```

**なぜ Group が要るのか**  
個人の論文マップだけでは、ゼミで「誰が何を読んだか」「関連図はどうなっているか」を見えにくいからです。  
Group は **閲覧の範囲を「ゼミメンバー」に限定する単位** として機能します。

#### `GroupsPage` / `GroupDetailPage` / `group_members` の対応

| 画面・コード | ユーザーから見える名前 | 裏で触るデータ |
|--------------|------------------------|----------------|
| `GroupsPage` | グループ一覧・作成・参加 | `groups`, `group_members` |
| `GroupDetailPage` | メンバー一覧・招待コード表示 | `groups`, `group_members`, `profiles` |
| `MemberGraphPage` | ○○さんの論文マップ | 他人の `nodes`, `edges`（閲覧のみ） |
| `FeedPage` | グループ内タイムライン | `posts`（`group_id` で紐づく） |

### 0.4 このアプリでよく出る「ものの名前」一覧

Group と並んで、コードや画面で繰り返し出る言葉です。

| 用語 | 何を指すか | DB テーブル |
|------|------------|-------------|
| **ユーザー / プロフィール** | ログインアカウント。表示名あり | `profiles`（Auth と連携） |
| **ノード（node）** | 論文や手法の 1 件。グラフ上の箱 | `nodes` |
| **エッジ（edge）** | ノード間の矢印・関係 | `edges` |
| **メモ（memo）** | ノードに付ける学習メモ | `memos` |
| **グループ（group）** | ゼミなどの共有単位 | `groups` + `group_members` |
| **投稿（post）** | フィードの 1 件。グラフのコピー付き | `posts` + `post_snapshots` |
| **マイグレーション** | DB の設計変更を記録した SQL ファイル | `supabase/migrations/*.sql` |
| **RLS** | 行ごとのアクセス権限ルール | 各テーブルの policy |
| **RPC** | DB 内の関数を API のように呼ぶこと | 例: `join_group_by_invite_code` |

#### ノードとエッジのイメージ（おさらい）

```
[Q-learning] ──→ [DQN] ──→ [PPO]
     ノード          エッジ      ノード
```

- **ノード** = 論文・手法の「点」
- **エッジ** = 「Q-learning が DQN の前提」のような「矢印」

#### マイグレーションとは

「DB の設計図の変更履歴」です。Git でコードの変更を追うように、DB の変更も SQL ファイルで追います。

```
20260531000000_initial_schema.sql   … 最初のテーブル作成
20260601000000_posts_feed.sql       … フィード機能追加
20260603000000_join_group_by_invite_code.sql … 招待参加の関数追加
```

`npm run db:reset` すると、これらが **番号順に全部実行** され、同じ DB 構造が再現されます。

### 0.5 `localhost` と `package.json` とは

## 1. Web アプリはブラウザの中で動く

普通の Web サイトもアプリも、大まかには同じです。

```
① ユーザーが URL を開く
② サーバーから HTML / JS / CSS が届く
③ ブラウザが JS を実行する
④ 画面が動く（ボタンを押す、データを取る、など）
```

Research Compass では **③ の JS がほぼ全部 React アプリ** です。  
サーバー側で HTML を毎回組み立てるのではなく、**ブラウザが空の箱（`<div id="root">`）に React が中身を描画** します。

```9:12:frontend/index.html
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
```

| 部分 | 役割 |
|------|------|
| `<div id="root">` | React が画面を描画する「キャンバス」 |
| `<script src="/src/main.tsx">` | ここからアプリが起動する |

**なぜこうするのか**  
1 つの HTML ファイルのままでは、画面が複雑になると管理できません。  
JS で画面を組み立てる仕組み（React）を使い、部品単位で開発します。

## 2. JavaScript / TypeScript の最低限

このプロジェクトのフロントは **TypeScript（.ts / .tsx）** です。  
JavaScript に **「この変数はこういう形のデータです」という約束（型）** を付けたものです。

> `.ts` と `.tsx` の違い、ブラウザがそのまま実行できない理由は [§0.1](#01-ファイルの拡張子-js--ts--jsx--tsx-とは何か) を参照。

### よく出る書き方

```typescript
// 変数
const name = 'Research Compass'   // 文字列（変えられない）
let count = 0                     // 数値（あとから変えられる）

// 関数
function add(a: number, b: number): number {
  return a + b
}

// オブジェクト（まとまったデータ）
const user = { id: 'abc', email: 'a@b.com' }

// 配列
const groups: Group[] = []

// import（他ファイルの機能を借りる）
import { useState } from 'react'
import type { Group } from '../types'
```

### `type` と `import type`

```typescript
import type { Group } from '../types'
```

- `Group` は **実行時には消える型情報だけ** を import している
- 「この変数は Group 型のオブジェクトであるべき」とコンパイラに伝える
- **なぜするのか**: タイポや間違ったプロパティ名を書く前にエディタが教えてくれる

### `export` と `import` の関係

| 書き方 | 意味 |
|--------|------|
| `export function GroupsPage()` | 他のファイルから使えるように公開 |
| `import { GroupsPage } from './pages/GroupsPage'` | 公開されたものを取り込む |

**ルール**: 1 ファイルに 1 つのメインコンポーネントを `export` するのがこのプロジェクトの慣習です。

## 3. async / await（非同期処理）

DB からデータを取る処理は **待ち時間** があります。  
その間、画面を固めないために **非同期（async）** で書きます。

```typescript
// 普通の関数 — すぐ終わる
function greet() {
  return 'こんにちは'
}

// 非同期関数 — 待つ処理を含む
async function load() {
  const list = await fetchGroups()  // 終わるまでここで待つ
  setGroups(list)                   // 終わったら state を更新
}
```

| キーワード | 意味 |
|------------|------|
| `async` | この関数の中では「待ち」が発生しうる |
| `await` | 右側の処理が終わるまで一時停止 |
| `Promise` | 「あとで結果が返ってくる約束」 |

### `void` を付ける理由

```typescript
onSubmit={(e) => void handleJoin(e)}
```

- `handleJoin` は `async` なので戻り値は `Promise`
- イベントハンドラの戻り値は通常使わない
- `void` は「戻り値は意図的に捨てる」という明示
- **なぜ**: ESLint が「await 忘れ」の警告を出さないようにするため

### try / catch / finally

```typescript
try {
  await createGroup(...)
  await load()           // 成功したら一覧を再取得
} catch (err) {
  setError(...)          // 失敗したらエラーメッセージ表示
} finally {
  setSubmitting(false)   // 成功・失敗どちらでも「送信中」を解除
}
```

**なぜ finally があるのか**  
ネットワークエラーでも、必ず「ボタンを再度押せる状態」に戻すためです。

## 5. React（画面を部品で組み立てる）

### React とは

**UI を「コンポーネント」という関数単位の部品に分けて組み立てるライブラリ** です。

```tsx
// これがコンポーネント（部品）
export function GroupsPage() {
  return (
    <section>
      <h1>グループ</h1>
    </section>
  )
}
```

### JSX とは

`return (...)` の中の **HTML に似た記法** です。実際は JavaScript です。

```tsx
<h1 className="text-2xl">グループ</h1>
```

| HTML | JSX（React） | 理由 |
|------|--------------|------|
| `class="..."` | `className="..."` | `class` は JS の予約語と被る |
| `for="..."` | `htmlFor="..."` | 同様に `for` が予約語 |

### コンポーネントの呼び出し方

```tsx
<GraphEditor userId={user.id} readOnly />
```

これは **「GraphEditor という部品を使う」** という意味です。

| 部分 | 名前 | 意味 |
|------|------|------|
| `userId={user.id}` | props（プロパティ） | 親から子へ渡すデータ |
| `readOnly` | boolean の省略記法 | `readOnly={true}` と同じ |

子コンポーネント側では引数として受け取ります。

```typescript
type GraphEditorProps = {
  userId: string
  readOnly?: boolean    // ? は「なくてもよい」
  targetUserId?: string
}

export function GraphEditor({ userId, readOnly = false, targetUserId }: GraphEditorProps) {
  ...
}
```

### React の再描画の仕組み（超重要）

```
state が変わる
    ↓
そのコンポーネントの関数がもう一度実行される
    ↓
return の JSX が新しい内容になる
    ↓
ブラウザの表示が更新される
```

**だから** 「変わる値」は `useState` で持つ必要があります。  
普通の `let x = 0` では、画面は更新されません。

### `{...}` で JS を埋め込む

```tsx
{loading ? <p>読み込み中...</p> : <ul>...</ul>}
{error && <p>{error}</p>}
{groups.map((group) => <li key={group.id}>...</li>)}
```

| 記法 | 意味 |
|------|------|
| `{変数}` | 変数の値を表示 |
| `{条件 ? A : B}` | 三項演算子。if の短い書き方 |
| `{error && <p>...}` | error が truthy なら右を表示 |
| `.map(...)` | 配列の各要素を JSX に変換 |
| `key={group.id}` | リストの各要素を React が識別するための ID（必須） |

**`key` のルール**: リストでは安定した一意の値（DB の `id` など）を使う。配列の index は並び替え時にバグの元。

## 6. React Router（URL と画面の切り替え）

### 何をするか

**URL（アドレスバーの `/graph` など）に応じて、表示するコンポーネントを切り替える** ライブラリです。  
ページ全体をサーバーに取りに行くのではなく、**同じ React アプリの中で画面だけ差し替え** ます（SPA = Single Page Application）。

### 基本部品

| 部品 | 役割 |
|------|------|
| `BrowserRouter` | ルーティング機能をアプリ全体に提供 |
| `Routes` / `Route` | URL パスとコンポーネントの対応表 |
| `Link` | ページ遷移するリンク（`<a>` の代わり） |
| `Navigate` | プログラムから別 URL へリダイレクト |
| `Outlet` | ネストした子ルートの表示場所 |
| `useNavigate()` | コードから遷移（ログイン後など） |
| `useParams()` | URL の `:id` 部分を取得 |
| `useLocation()` | 今の URL 情報 |

### ネスト（入れ子）の意味

```tsx
<Route element={<ProtectedRoute />}>
  <Route element={<AppLayout />}>
    <Route path="/groups" element={<GroupsPage />} />
  </Route>
</Route>
```

描画のイメージ:

```
ProtectedRoute がログイン確認
  → OK なら <Outlet /> の位置に AppLayout
      → AppLayout がヘッダー + <Outlet />
          → <Outlet /> の位置に GroupsPage
```

**なぜネストするのか**  
「ログイン必須」「共通ヘッダー」を **各ページごとに書かずに 1 回だけ** 適用できるからです。

### `Link` の使い方

```tsx
<Link to={`/groups/${group.id}`}>...</Link>
```

- `` `/groups/${group.id}` `` はテンプレートリットラル（文字列の中に変数を埋め込む）
- クリックすると URL が変わり、対応する `Route` のコンポーネントが表示される
- **通常の `<a href>` との違い**: ページ全体の再読み込みが起きない（速い）

## 8. Supabase JS Client（DB・認証との通信）

### 何をするか

ブラウザから **Supabase クラウド（またはローカル Supabase）** に HTTP でアクセスするためのライブラリです。

```typescript
import { createClient } from '@supabase/supabase-js'

export const supabase = createClient(supabaseUrl, supabaseAnonKey)
```

一度 `createClient` したオブジェクトを **アプリ全体で共有** します。

### よく使うメソッド

| コード | やっていること | SQL に例えると |
|--------|----------------|----------------|
| `.from('nodes').select('*')` | 全件取得 | `SELECT * FROM nodes` |
| `.eq('user_id', id)` | 条件追加 | `WHERE user_id = id` |
| `.insert({...})` | 行を追加 | `INSERT INTO ...` |
| `.update({...}).eq('id', id)` | 行を更新 | `UPDATE ... WHERE id = ...` |
| `.delete().eq('id', id)` | 行を削除 | `DELETE ... WHERE id = ...` |
| `.rpc('関数名', { 引数 })` | DB の SQL 関数を呼ぶ | `SELECT 関数名(...)` |
| `supabase.auth.signInWithPassword` | ログイン | Auth 専用 API |

### 返り値の形

```typescript
const { data, error } = await supabase.from('groups').select('*')
if (error) throw error
return data ?? []
```

| 部分 | 意味 |
|------|------|
| `data` | 成功時の結果 |
| `error` | 失敗時のエラー情報 |
| `data ?? []` | data が null/undefined なら空配列を使う |

**なぜ `throw error` するのか**  
呼び出し元の `try/catch` でまとめてエラー表示できるようにするため（`GroupsPage` の `handleCreate` など）。

### 認証トークンは自動

ログイン後、Supabase クライアントが **JWT トークンを自動でリクエストに付ける** ので、  
`fetchNodes` などは `userId` をクエリに書くだけで、DB 側は `auth.uid()` で本人確認できます。

## 9. React Flow（グラフ描画）

### 何をするか

**ノード（箱）とエッジ（矢印）のグラフを、ドラッグ・ズーム・接続できる UI として描画する** ライブラリです。  
論文マップの「地図」部分はほぼ React Flow のお世話です。

### 基本概念

| 用語 | React Flow での意味 |
|------|---------------------|
| Node | 画面上の 1 つの箱（論文） |
| Edge | 箱と箱を結ぶ線 |
| Handle | 線をつなぐ接続点（`PaperNode.tsx` の四辺） |
| Viewport | ズーム・パンの表示領域 |

### このプロジェクトでの使い方

```tsx
<ReactFlow
  nodes={nodes}
  edges={edges}
  onNodesChange={onNodesChange}
  onNodeDragStop={onNodeDragStop}
  onConnect={onConnect}
  nodeTypes={nodeTypes}
  edgeTypes={edgeTypes}
/>
```

| props | 役割 |
|-------|------|
| `nodes` / `edges` | 今表示するデータ |
| `onNodesChange` | ドラッグなどの変化を React Flow に伝える（必須） |
| `onNodeDragStop` | ドラッグ終了時に DB へ座標保存 |
| `onConnect` | 2 ノードを線で結んだとき |
| `nodeTypes` | ノードの見た目をカスタム（`PaperNodeComponent`） |
| `edgeTypes` | エッジの見た目をカスタム（`MidpointEdge`） |

### `useNodesState` / `useEdgesState`

React Flow 専用の state フックです。  
ドラッグ中の一時的な位置変更などをスムーズに処理するために使います。

## 10. コンポーネントの書き方ルール

### ルール一覧

| ルール | 内容 |
|--------|------|
| 1 ファイル 1 コンポーネント（目安） | `GroupsPage.tsx` → `export function GroupsPage` |

### コンポーネントの中身の典型構造

```tsx
export function SomePage() {

  // ② データ取得・イベント処理の関数
```

## 11. useState — 画面が覚えておく値

### 書き方

```typescript
const [groups, setGroups] = useState<GroupWithCount[]>([])
const [loading, setLoading] = useState(true)
const [name, setName] = useState('')
```

| 部分 | 意味 |
|------|------|
| `groups` | 今の値（読むとき） |
| `setGroups` | 値を変える関数（書くとき） |
| `useState([])` | 初期値は空配列 |
| `useState(true)` | 初期値は「読み込み中」 |

### 更新のしかた

```typescript
// 完全に置き換え
setGroups(withCounts)

// 前の値を使って更新（関数形式）
setPosts((prev) => [post, ...prev])
```

**なぜ `(prev) => ...` を使うのか**  
複数の更新が続くとき、古い state を参照してしまうバグを防ぐためです。

### 入力フォームとの結びつき

```tsx
<input
  value={name}
  onChange={(e) => setName(e.target.value)}
/>
```

| 部分 | 流れ |
|------|------|
| `value={name}` | 画面に表示する文字は state の `name` |
| `onChange` | ユーザーが打つたびに `setName` で state 更新 |
| 再描画 | 新しい `name` が input に反映される |

これを **制御コンポーネント（Controlled Component）** と呼びます。  
**ルール**: フォームの値は React の state が「唯一の正」になるようにつなぐ。

## 12. useEffect — タイミングよく処理を走らせる

### 何のためのフックか

**「画面が表示されたとき」「特定の値が変わったとき」に、決まった処理を実行する** ためです。

```typescript
useEffect(() => {
  void load()
}, [])
```

| 部分 | 意味 |
|------|------|
| 第 1 引数 `() => { ... }` | 実行したい処理 |
| 第 2 引数 `[]` | 依存配列。「いつ再実行するか」 |

### 依存配列のパターン

| 依存配列 | 実行タイミング |
|----------|----------------|
| `[]` | 最初の 1 回だけ（マウント時） |
| `[ownerId]` | 最初 + `ownerId` が変わるたび |
| 省略 | 毎回の再描画のたび（ほぼ使わない） |

### GroupsPage での使い方

```45:48:frontend/src/pages/GroupsPage.tsx
  useEffect(() => {
    void load()
  }, [])
```

**なぜここで load するのか**  
ページを開いた瞬間にグループ一覧を DB から取ってこないと、画面が空のままだからです。

**なぜ関数の中に直接書かないのか**  
レンダリングのたびに DB アクセスが走ると無限ループや無駄な通信になるからです。  
`useEffect` は「副作用（サーバー通信など）」をレンダリングと分離する場所です。

### クリーンアップ（応用）

```typescript
useEffect(() => {
  const sub = supabase.auth.onAuthStateChange(...)
  return () => sub.unsubscribe()  // コンポーネント破棄時に解除
}, [])
```

**なぜ**: メモリリークや二重登録を防ぐため。

## 13. Context + カスタム Hook — 全体共有

### 問題：ログイン情報をどこでも使いたい

各ページで毎回 `supabase.auth.getSession()` するのは大変です。  
**アプリのルートで 1 回だけ管理し、どこからでも取り出す** 仕組みが Context です。

### 3 ファイルに分かれている理由

```
auth-context.ts   … Context の「型」と入れ物を定義
AuthContext.tsx   … Provider（値を実際に入れて配る）
useAuth.ts        … 取り出す用の Hook
```

#### ① auth-context.ts — 入れ物の定義

```1:17:frontend/src/contexts/auth-context.ts
export type AuthContextValue = {
  user: User | null
  session: Session | null
  loading: boolean
  signIn: (email: string, password: string) => Promise<{ error: string | null }>
  signUp: (...)
  signOut: () => Promise<void>
}

export const AuthContext = createContext<AuthContextValue | null>(null)
```

- `createContext` で「グローバルな入れ物」を作る
- 中身の形を `AuthContextValue` 型で決める

#### ② AuthContext.tsx — 値を入れて子に渡す

```tsx
// main.tsx でアプリ全体を包む
<AuthProvider>
  <App />
</AuthProvider>
```

`AuthProvider` の中で:

1. `useState` で `session` を保持
2. `useEffect` で Supabase からセッション取得・変更監視
3. `signIn` / `signOut` 関数を定義
4. `<AuthContext.Provider value={...}>` で子コンポーネントに配る

#### ③ useAuth.ts — 取り出す

```1:10:frontend/src/hooks/useAuth.ts
export function useAuth() {
  const context = useContext(AuthContext)
  if (!context) {
    throw new Error('useAuth must be used within AuthProvider')
  }
  return context
}
```

ページ側はシンプルに:

```typescript
const { user, signOut } = useAuth()
```

**なぜ Hook にするのか**  
`useContext(AuthContext)` を毎回書かず、`useAuth()` 1 行にまとめる。  
Provider の外で使ったらすぐエラーにする安全装置も付けられる。

## 14. フォームの書き方ルール

### 基本パターン

```tsx
<form onSubmit={(e) => void handleCreate(e)}>
  <input value={name} onChange={(e) => setName(e.target.value)} required />
  <button type="submit" disabled={submitting}>作成</button>
</form>
```

| ルール | 理由 |
|--------|------|
| `onSubmit` で送信処理 | Enter キーでも送信できる。`button` の click だけにしない |
| `e.preventDefault()` | ブラウザ標準の「ページ全体の再読み込み」を止める |
| `disabled={submitting}` | 二重送信を防ぐ |
| `required` | 簡易な未入力チェック |

### handleCreate の流れ（GroupsPage）

```
ユーザーが「作成」クリック
  → handleCreate 実行
  → e.preventDefault()
  → setSubmitting(true)     … ボタン無効化
  → setError(null)          … 前のエラーを消す
  → await createGroup(...)  … DB に保存
  → フォームをリセット
  → await load()            … 一覧を最新化
  → catch で setError
  → finally で setSubmitting(false)
```

**なぜ作成後に load() するのか**  
`createGroup` の返り値だけでは一覧の `member_count` などが揃っていない場合がある。  
一覧表示用のデータを **もう一度まとめて取り直す** のが確実だからです。

## 15. 条件分岐で画面を切り替えるパターン

### パターン① 早期 return（ガード）

```typescript
if (!user) return null
```

ログイン前は何も描画しない。`ProtectedRoute` で弾かれる前提の二重チェックです。

### パターン② 三項演算子

```tsx
{loading ? (
  <p>読み込み中...</p>
) : groups.length === 0 ? (
  <p>グループがありません</p>
) : (
  <ul>...</ul>
)}
```

**読み方**: loading なら A、そうでなくて空なら B、それ以外は C。

### パターン③ && による部分表示

```tsx
{error && <p className="...">{error}</p>}
{showCreate ? <form>...</form> : <p>新しいゼミグループを...</p>}
```

`error` が `null` のときは何も出さない。エラーがあるときだけ赤い枠を表示。

## 16. 例題① index.html → main.tsx（起動の流れ）

### ステップ全体

```
index.html
  → main.tsx が読み込まれる
  → createRoot(...).render(...) で React 起動
  → AuthProvider が App を包む
  → App.tsx の Routes が URL に応じたページを選ぶ
```

### main.tsx 1 行ずつ

```tsx
createRoot(document.getElementById('root')!).render(
```

| 部分 | 説明 |
|------|------|
| `document.getElementById('root')` | HTML の `<div id="root">` を取得 |
| `!` | TypeScript に「null ではない」と伝える |
| `createRoot(...).render(...)` | ここに React の木を描画 |

```tsx

    <AuthProvider>
```

| 部分 | 説明 |
|------|------|

## 17. 例題② useAuth / AuthContext（認証の仕組み）

### ログイン成功までの流れ

```
LoginPage: ユーザーがメール・パスワード入力
  → signIn(email, password)  [AuthContext 内]
  → supabase.auth.signInWithPassword(...)
  → Supabase が JWT を返す
  → onAuthStateChange が発火
  → setSession(nextSession)
  → user が入った状態で再描画
  → ProtectedRoute が通過
  → 元のページへ navigate
```

### loading が必要な理由

```typescript
const [loading, setLoading] = useState(Boolean(supabase))
```

ページを開いた直後は **まだセッション確認中** です。  
この間に `user` が null だと、一瞬だけログイン画面に飛ばされる（ちらつき）ので、

```tsx
if (loading) return <div>読み込み中...</div>
```

と待ちます。

### signUp で display_name を渡す理由

```typescript
await supabase.auth.signUp({
  email,
  password,
  options: { data: { display_name: displayName } },
})
```

Supabase Auth の `raw_user_meta_data` に入り、DB トリガー `handle_new_user` が `profiles.display_name` にコピーします。

## 18. 例題③ GroupsPage.tsx を上から読む

> この画面で扱う「グループ」がアプリ上で何を意味するかは [§0.3](#03-groupグループとは何か--このアプリでの意味) を参照。

### 1〜7 行目：import

```typescript
import { type FormEvent, useEffect, useState } from 'react'
```

| import | 用途 |
|--------|------|
| `FormEvent` | フォーム送信イベントの型 |
| `useEffect` | 初回データ取得 |
| `useState` | 画面の状態管理 |

```typescript
import { createGroup, fetchGroupMembers, fetchGroups, joinGroupByCode } from '../lib/api/groups'
```

**ルール**: ページは `lib/api` の関数を呼ぶだけ。SQL は書かない。

### 7 行目：型の拡張

```typescript
type GroupWithCount = Group & { member_count: number }
```

- `Group` 型に `member_count` を足した型
- `&` は型の合成（「Group かつ member_count を持つ」）
- **なぜ**: DB の `groups` テーブルには `member_count` 列はない。画面用に後から計算して付けるから

### 18〜27 行目：state 一覧

| state | 初期値 | 役割 |
|-------|--------|------|
| `groups` | `[]` | 表示するグループ一覧 |
| `loading` | `true` | 読み込み中か |
| `showCreate` | `false` | 作成フォームを開いているか |
| `name` | `''` | 作成フォームの名前入力 |
| `description` | `''` | 説明入力 |
| `inviteCode` | `''` | 参加コード入力 |
| `error` | `null` | エラーメッセージ |
| `submitting` | `false` | 送信中か（ボタン無効用） |

**なぜこんなに state を分けるのか**  
1 つの大きなオブジェクトでもよいが、**入力ごとに独立して更新** した方がフォームは書きやすいからです。

### 29〜43 行目：load 関数

```typescript
const list = await fetchGroups()
const withCounts = await Promise.all(
  list.map(async (g) => {
    const members = await fetchGroupMembers(g.id)
    return { ...g, member_count: members.length }
  }),
)
setGroups(withCounts)
```

| コード | 説明 |
|--------|------|
| `fetchGroups()` | 自分が所属するグループ一覧 |
| `list.map(async (g) => ...)` | 各グループごとにメンバー数を取得 |
| `{ ...g, member_count: ... }` | スプレッド構文。g の全フィールドをコピーしつつ `member_count` を追加 |
| `Promise.all(...)` | 全部の非同期処理が終わるまで待つ |

**なぜメンバー数を別クエリで取るのか**  
MVP では SQL の JOIN をフロント側で使わず、**分かりやすさ優先** で 2 段階取得にしているからです。

### 105〜120 行目：参加フォーム

```tsx
onChange={(e) => setInviteCode(e.target.value.toUpperCase())}
```

**なぜ toUpperCase() するのか**  
DB 側の `join_group_by_invite_code` が `upper(trim(...))` で比較するため。  
ユーザーが小文字で入れても一致するように、入力時に大文字化しています。

### 188〜209 行目：リスト表示

```tsx
{groups.map((group) => (
  <li key={group.id}>
    <Link to={`/groups/${group.id}`}>...</Link>
  </li>
))}
```

- 配列 `groups` の要素ごとに `<li>` を 1 つ生成
- `key={group.id}` は React が「どの行が変わったか」を判別するため必須
- `Link to={...}` でグループ詳細ページへ

## 19. 例題④ lib/api/groups.ts（DB への問い合わせ）

### fetchGroups

```typescript
const { data, error } = await supabase
  .from('groups')
  .select('*')
  .order('created_at', { ascending: false })
```

- RLS により **自分がメンバーのグループだけ** 返る
- フロントで `WHERE user_id = ...` を書いていないのは、**DB が権限で絞っている** から

### createGroup が 2 回 insert する理由

```typescript
// 1. groups テーブルにグループ本体
.insert({ name, invite_code, created_by: userId })

// 2. group_members に自分を追加
.insert({ group_id: data.id, user_id: userId })
```

**なぜ 1 回でやらないのか**  
テーブルが分かれている（グループ情報 / メンバー関係）。  
作成者もメンバーに含めないと、RLS で「自分のグループ」として見えなくなるからです。

### joinGroupByCode が rpc を使う理由

```typescript
const { data: group, error } = await supabase.rpc('join_group_by_invite_code', {
  p_invite_code: inviteCode.trim(),
})
```

普通の `select` では未参加のグループ行が RLS で見えません。  
**DB 内の特権関数** が招待コードで検索し、メンバー追加まで行います。

フロントはエラーメッセージを日本語化:

```typescript
if (msg.includes('INVITE_CODE_NOT_FOUND')) {
  throw new Error('招待コードが見つかりません')
}
```

**なぜ DB の英語エラーを変換するのか**  
ユーザー向け画面には分かりやすい日本語を出すため。  
`throw new Error(...)` すると `GroupsPage` の `catch` で `setError` できます。

## 20. 例題⑤ GraphEditor の state が 2 段ある理由

GraphEditor には似た名前の state が 2 種類あります。

| state | 型 | 役割 |
|-------|-----|------|
| `dbNodes` / `dbEdges` | `PaperNode[]` / `PaperEdge[]` | **DB と一致する正本** |
| `nodes` / `edges` | React Flow 用 | **画面上の描画用**（ドラッグ中の一時位置など含む） |

### データの流れ

```
① loadGraph → fetchNodes/fetchEdges → dbNodes, dbEdges に保存
② useMemo で dbNodes → flowNodes に変換（toFlowNodes）
③ useEffect で flowNodes → nodes（React Flow 用 state）にコピー
④ ユーザーがドラッグ → React Flow が nodes を一時更新
⑤ ドラッグ終了 onNodeDragStop → updateNode で DB 保存 → dbNodes 更新
⑥ ②に戻って画面と DB が再同期
```

**なぜ 1 つにまとめないのか**

1. **DB の形**（`position_x`, `position_y`）と **React Flow の形**（`position: {x,y}`）が違う
2. React Flow はドラッグ中に `onNodesChange` で細かく state を更新する。**その都度 DB に書くと重い**
3. 「正本は DB」「表示は React Flow」に分けると、読み取り専用モードやフィルタ（関連のみ表示）も `useMemo` で派生できる

### readOnly のとき

```tsx
onNodesChange={readOnly ? undefined : onNodesChange}
nodesDraggable={!readOnly}
```

`undefined` を渡すとその操作が無効になります。  
**同じコンポーネント** で編集・閲覧を切り替えるためのスイッチです。

## 23. よくある「なぜ？」まとめ

| 質問 | 答え |
|------|------|
| `.tsx` とは何か | TypeScript + JSX。画面用のファイル形式。ブラウザは読めないので Vite が JS に変換する（[§0.1](#01-ファイルの拡張子-js--ts--jsx--tsx-とは何か)） |
| 開発サーバーとは何か | 開発中だけ PC 上で動く「変換して配る係」。`npm run dev` で起動（[§0.2](#02-開発サーバーとビルドとは何か)） |
| ビルドとは何か | 本番用に `dist/` を一括生成すること。`npm run build`。Render はこの成果物を配信 |
| Group とは何か | ゼミなどの共有単位。`groups` + `group_members` テーブルで表現。メンバー同士が論文マップを閲覧できる範囲（[§0.3](#03-groupグループとは何か--このアプリでの意味)） |
| なぜ React を使うのか | 画面を部品化し、状態（state）と連動させやすいから。論文グラフのような複雑 UI 向き |
| なぜ API サーバーを自前で書かないのか | Supabase が Auth + DB + API を提供。1 週間 MVP では工数削減 |
| なぜ RLS が必要か | anon key はフロントに露出する。DB 側で「他人のデータは読めない」を強制しないと危険 |
| なぜ TypeScript か | 型でミスを早く発見。`PaperNode` の形が docs 代わりになる |
| なぜ `useEffect` で fetch するのか | 描画のたびに fetch するとループ・無駄通信になるから |
| なぜ Context で認証か | どのページでも `user` が要る。props で何層も渡すのは面倒 |
| なぜ GraphEditor を共有するのか | 編集と閲覧は UI が 90% 同じ。`readOnly` で差を吸収 |
| なぜ post_snapshots に JSON コピーか | 投稿時点の地図を残すため。あとから本人がグラフを変えても投稿は変わらない |
| なぜ RPC でグループ参加か | RLS の都合で未参加グループを普通の SELECT では見つけられないから |
| `void load()` とは | async 関数の Promise を意図的に無視して ESLint を満たす書き方 |
