## 1. このアプリは何をするものか

**Research Compass** は、論文や研究手法を「地図（グラフ）」として可視化する Web アプリです。

### ユーザーができること

| 機能 | 説明 |
|------|------|
| 論文グラフの作成 | 論文・手法を「ノード（点）」として追加し、矢印（エッジ）で関係を結ぶ |
| メモ・理解度の管理 | 各ノードに「わからない点」「メモ」「理解度（◎○△×）」を保存 |
| 関連/非関連の切り替え | 今の研究テーマと関係ない論文は「非関連」にして薄く表示 |
| グループ共有 | ゼミなどのグループを作り、メンバー同士で各自の論文図を**閲覧**できる |
| フィード投稿 | グループ内に「今の論文マップのスナップショット付き」で投稿できる |

### やらないこと（MVP の範囲外）

- 論文 PDF の自動解析
- AI によるロードマップ自動生成
- グループ全体のグラフ自動マージ

## 2. 全体の枠組み（アーキテクチャ）

このアプリは **「フロントエンドだけのアプリ + バックエンドとして Supabase を借りる」** という形です。

```
┌─────────────────────────────────────────────────────────────┐
│  ブラウザ（ユーザーが見る画面）                               │
│  React アプリ（frontend/）                                    │
│    - 画面表示（ページ・コンポーネント）                        │
│    - グラフ描画（React Flow）                                 │
│    - Supabase へのデータ取得・保存                            │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS（インターネット経由）
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Supabase（クラウド上の BaaS = Backend as a Service）        │
│    ├── Auth      … ログイン・ユーザー管理（JWT トークン発行）  │
│    ├── PostgreSQL … データ保存（nodes, edges, groups など）  │
│    └── RLS       … 「誰が何を読めるか」を DB 側で強制         │
└─────────────────────────────────────────────────────────────┘
```

### ポイント：自前のサーバー（FastAPI など）を書いていない

一般的な Web アプリでは「フロント」と「バックエンド API サーバー」を別々に作ります。  
このプロジェクトでは **Supabase がバックエンドの役割を担う** ため、フロントから直接 DB にアクセスする形になっています。

- **メリット**: 認証・DB・API が最初から揃っている。1 週間のハッカソン向き
- **注意点**: セキュリティは **RLS（Row Level Security）** で DB 側に任せる必要がある

### 本番環境の構成

| 役割 | サービス |
|------|----------|
| フロントのホスティング | [Render](https://research-compass-7o92.onrender.com/)（静的サイト） |
| バックエンド | [Supabase Cloud](https://supabase.com/) |

## 5. データの流れ（1回の操作で何が起きるか）

例：**ユーザーが論文ノードを追加する**

```
① ユーザーが「+ ノード追加」ボタンを押し、タイトルを入力して送信
        ↓
② GraphEditor.tsx の AddNodeButton が handleSubmit を実行
        ↓
③ lib/api/nodes.ts の createNode(userId, { title, position_x, ... }) を呼ぶ
        ↓
④ supabase.from('nodes').insert({...}) が HTTP リクエストを送る
        ↓
⑤ Supabase が JWT を検証し、RLS で「自分の user_id なら INSERT 可」を確認
        ↓
⑥ PostgreSQL の nodes テーブルに 1 行追加
        ↓
⑦ 追加された行が JSON で返る
        ↓
⑧ React の state（dbNodes）が更新され、画面のグラフに新ノードが表示される
```

**重要**: フロントは「表示」と「ユーザーの操作を受け取る」役。  
**データの正しさ・権限チェックは DB（RLS）側** が最終的に守ります。

## 7. 詳細解説：認証（ログイン・ログアウト）

### 7.1 認証の仕組み（概念）

1. ユーザーがメール＋パスワードを送信
2. Supabase Auth が照合し、成功すると **JWT（トークン）** を発行
3. ブラウザがトークンを保持（Supabase クライアントが自動管理）
4. 以降の DB リクエストにはトークンが付く → DB は `auth.uid()` で「誰か」を判別

### 7.2 Supabase 接続設定

```1:13:frontend/src/lib/supabase.ts
import { createClient } from '@supabase/supabase-js'

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY

export const supabase =
  supabaseUrl && supabaseAnonKey
    ? createClient(supabaseUrl, supabaseAnonKey)
    : null

export function isSupabaseConfigured(): boolean {
  return supabase !== null
}
```

- `VITE_` で始まる環境変数は Vite がビルド時に埋め込む
- `anon key` は「公開してよいキー」。ただし RLS があるので、キーだけでは他人のデータは読めない

### 7.3 AuthContext（ログイン状態の共有）

```6:69:frontend/src/contexts/AuthContext.tsx
export function AuthProvider({ children }: { children: ReactNode }) {
  const [session, setSession] = useState<Session | null>(null)
  const [loading, setLoading] = useState(Boolean(supabase))

  useEffect(() => {
    if (!supabase) return

    supabase.auth.getSession().then(({ data }) => {
      setSession(data.session)
      setLoading(false)
    })

    const {
      data: { subscription },
    } = supabase.auth.onAuthStateChange((_event, nextSession) => {
      setSession(nextSession)
      setLoading(false)
    })

    return () => subscription.unsubscribe()
  }, [])

  const signIn = useCallback(async (email: string, password: string) => {
    ...
    const { error } = await supabase.auth.signInWithPassword({ email, password })
    ...
  }, [])
  ...
}
```

**やっていること**

| 処理 | 説明 |
|------|------|
| `getSession()` | ページ読み込み時、保存済みセッションを復元 |
| `onAuthStateChange` | ログイン・ログアウト時に自動で state 更新 |
| `signIn` / `signUp` / `signOut` | 各画面から呼べる認証関数 |

### 7.4 ログイン画面の流れ

```22:33:frontend/src/pages/LoginPage.tsx

    e.preventDefault()
```

1. フォーム送信 → `signIn` 呼び出し
2. エラーなら画面に表示
3. 成功なら元のページ（または `/`）へ遷移

### 7.5 新規登録時のプロフィール自動作成

ユーザー登録すると、DB 側のトリガーが `profiles` テーブルに行を自動追加します。

```94:113:supabase/migrations/20260531000000_initial_schema.sql

returns trigger
language plpgsql
security definer

as $$
begin
  insert into public.profiles (id, email, display_name)
```

## 8. 詳細解説：Supabase クライアントと API 層

### 8.1 なぜ API 層（lib/api/）があるのか

ページのコンポーネントから直接 `supabase.from(...)` を書くこともできますが、  
**DB 操作を `lib/api/` に集約** することで:

- 同じ処理の再利用がしやすい
- テーブル名・カラム名の変更が 1 箇所で済む
- ページ側は「何をしたいか」だけ読める

### 8.2 ノード取得・作成の例（nodes.ts）

```4:38:frontend/src/lib/api/nodes.ts
export async function fetchNodes(userId?: string): Promise<PaperNode[]> {
  if (!supabase) return []
  let query = supabase.from('nodes').select('*').order('created_at')
  if (userId) query = query.eq('user_id', userId)
  const { data, error } = await query
  if (error) throw error
  return (data ?? []).map(normalizeNode)
}

export async function createNode(
  userId: string,
  input: PaperNodeInput,
): Promise<PaperNode> {
  if (!supabase) throw new Error('Supabase が未設定です')
  const { data, error } = await supabase
    .from('nodes')
    .insert({
      user_id: userId,
      title: input.title,
      ...
    })
    .select()
    .single()
  if (error) throw error
  return normalizeNode(data)
}
```

**Supabase クエリの読み方**

| コード | 意味 |
|--------|------|
| `.from('nodes')` | `nodes` テーブルを対象にする |
| `.select('*')` | 全カラム取得（SQL の `SELECT *`） |
| `.eq('user_id', userId)` | `WHERE user_id = ?` |
| `.insert({...})` | 行を追加 |
| `.select().single()` | 追加した行を 1 件返してもらう |

### 8.3 型定義（types/index.ts）

TypeScript の型は「データの形の約束」です。フロントと DB の橋渡しになります。


## 9. 詳細解説：論文グラフ（React Flow）

### 9.1 グラフ画面の構成

```
GraphPage.tsx          … ページの枠（タイトル表示）
  └── GraphEditor.tsx  … グラフ本体（読み込み・編集・表示すべて）
        ├── ReactFlow  … ライブラリによるキャンバス
```

## 11. 詳細解説：グループ機能と権限制御（RLS）

### 11.1 グループ機能でできること

| 操作 | 実装場所 |
|------|----------|
| グループ作成 | `createGroup()` |
| 招待コードで参加 | `joinGroupByCode()` → DB の RPC |
| メンバー一覧 | `fetchGroupMembers()` |
| 退出 | `leaveGroup()` → DB の RPC |
| 削除（オーナーのみ） | `deleteGroup()` |

### 11.2 グループ作成の流れ

```26:44:frontend/src/lib/api/groups.ts
export async function createGroup(
  userId: string,
  name: string,
  description: string,

  ...
  const invite_code = generateInviteCode()
  const { data, error } = await supabase
```

2 ステップです:

1. `groups` テーブルにグループ行を追加（ランダムな `invite_code` 付き）
2. 作成者を `group_members` に追加（自分もメンバーになる）

### 11.3 招待コード参加（RPC を使う理由）

通常の `SELECT` では、**まだ参加していないグループは RLS で見えない** ため、招待コードからグループを探せません。  
そこで DB 側に特権関数 `join_group_by_invite_code` を用意しています。

```4:36:supabase/migrations/20260603000000_join_group_by_invite_code.sql
create or replace function public.join_group_by_invite_code(p_invite_code text)
returns public.groups
language plpgsql
security definer
set search_path = public
set row_security = off
as $$
declare
  v_group public.groups;
  v_uid uuid := auth.uid();
begin
  ...
  select * into v_group
  from public.groups g
  where g.invite_code = upper(trim(p_invite_code));

  if not found then
    raise exception 'INVITE_CODE_NOT_FOUND';
  end if;

  insert into public.group_members (group_id, user_id)
  values (v_group.id, v_uid);
  ...
  return v_group;
end;
$$;
```

- `security definer` … 関数の所有者権限で実行（RLS を一時的にバイパス）
- `row_security = off` … この関数内では RLS を無効化
- フロントは `supabase.rpc('join_group_by_invite_code', { p_invite_code: ... })` で呼ぶ

### 11.4 RLS（Row Level Security）とは

PostgreSQL の機能で、**行ごとに「誰が読める・書けるか」** を SQL で定義します。

例：自分のノードだけ編集可能

```216:220:supabase/migrations/20260531000000_initial_schema.sql
create policy "Users manage own nodes"
  on public.nodes for all
  to authenticated
  using (user_id = auth.uid())
  with check (user_id = auth.uid());
```

| 句 | 意味 |
|----|------|
| `for all` | SELECT / INSERT / UPDATE / DELETE すべて |
| `to authenticated` | ログイン済みユーザーのみ |
| `using (...)` | 読む・削除するときの条件 |
| `with check (...)` | 書き込むときの条件 |
| `auth.uid()` | 今ログインしているユーザーの UUID |

### 11.5 グループメンバーは他人のグラフを「読める」

```222:225:supabase/migrations/20260531000000_initial_schema.sql
create policy "Group members can view others nodes"
  on public.nodes for select
  to authenticated
  using (public.are_group_members(auth.uid(), user_id));
```

`are_group_members(A, B)` は「A と B が同じグループにいるか」を調べる関数です。  
**編集は自分のデータだけ、閲覧は同じグループなら OK** というルールが DB で強制されます。

### 11.6 RLS の無限再帰問題と修正

`group_members` テーブルのポリシー内で、また `group_members` を参照すると無限ループになることがあります。  
修正版では `is_group_member()` という **RLS をオフにしたヘルパー関数** 経由でチェックします。

```55:58:supabase/migrations/20260602000000_fix_rls_group_members_recursion.sql
create policy "Members can view group membership"
  on public.group_members for select
  to authenticated
```

## まとめ：このプロダクトの設計思想

1. **フロントにロジックを寄せすぎない** … DB 操作は `lib/api/`、権限は RLS
2. **1 コンポーネントの使い回し** … `GraphEditor` が自分用・メンバー閲覧用の両方
3. **データと表示の分離** … `PaperNode`（DB）と `PaperFlowNode`（React Flow）を `graph-utils` で変換
4. **複雑な権限は SQL 関数へ** … 招待参加・退出は RPC で明示的なエラーメッセージ
5. **ハッカソン向きの割り切り** … 自前 API サーバーなし、Supabase に寄せる
